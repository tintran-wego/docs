# Repositories

This page lists all repositories owned and maintained by the Flight Shopping team.

## Core Repositories

### curiosity

**Main search orchestrator**

| | |
|---|---|
| **URL** | [github.com/wego/curiosity](https://github.com/wego/curiosity) |
| **Language** | Java |
| **Framework** | Dropwizard |
| **Deploy** | Jenkins → ECS |

<Accordion title="Key Information">
**Purpose**: Entry point for all flight searches. Orchestrates provider queries, aggregates results, and serves to frontend/partners.

**Main modules**:
- `curiosity-core`: Shared models and utilities
- `curiosity-providers`: Main application

**Quick start**:
```bash
git clone git@github.com:wego/curiosity.git
cd curiosity
asdf install
docker-compose up -d live-redis static-redis elasticmq
./gradlew clean build -x test
```

**Key contacts**: Entire Flight Shopping team
</Accordion>

### flight-integrations

**Provider integration layer**

| | |
|---|---|
| **URL** | [github.com/wego/flight-integrations](https://github.com/wego/flight-integrations) |
| **Language** | Java |
| **Framework** | Dropwizard |
| **Deploy** | Jenkins → ECS |

<Accordion title="Key Information">
**Purpose**: Contains 400+ provider integrations. Each provider has its own module with client, transformer, and parser.

**Structure**:
```
flights/
├── emirates.com/
├── sabre.com/
├── amadeus.com/
└── ... (400+ providers)
```

**Adding a provider**:
1. Update `local.properties` to include new provider
2. Create module in `flights/` directory
3. Implement client extending base classes
4. Configure in Static Redis

**Key contacts**: Entire Flight Shopping team
</Accordion>

### flight-offer-engine

**Offer processing and caching**

| | |
|---|---|
| **URL** | [github.com/wego/flight-offer-engine](https://github.com/wego/flight-offer-engine) |
| **Language** | Java |
| **Framework** | Spring Boot |
| **Runtime** | GraalVM |
| **Deploy** | Jenkins → ECS |

<Accordion title="Key Information">
**Purpose**: Processes and caches flight offers. Serves cached data to search and GFS.

**Tech stack**:
- Spring Boot 3.x
- GraalVM 21
- ScyllaDB
- LocalStack (local dev)

**Quick start**:
```bash
git clone git@github.com:wego/flight-offer-engine.git
cd flight-offer-engine
./run_infra_local.sh  # Starts ScyllaDB, LocalStack
./gradlew clean build
docker-compose -f docker-compose.debug.yml up -d
```
</Accordion>

## Repository Standards

### Branch Protection

All core repos have:
- `main` branch protection
- Required PR reviews (1-2 approvals)
- Required CI checks
- No direct pushes

### CI/CD

Standard pipeline:
1. **Build**: Compile, run checks
2. **Test**: Unit tests, integration tests
3. **Analyze**: SonarQube scan
4. **Deploy Staging**: Automatic on merge
5. **Deploy Production**: Manual trigger

### Code Owners

Each repo has `CODEOWNERS` file defining reviewers:

```
# CODEOWNERS
* @wego/flight-shopping

# Specific areas
/flights/gfs/** @tin @senior-team
```
